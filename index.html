Let's look at a brief history of clientside libraries and frameworks

It's been almost 20 years since the release of the library that in many ways popularised JavaScript and turned it into what it is today. A few years later, we started seeing frameworks pop up, and a new era of the frontend began.

7 years after the launch of jQuery, Facebook open sourced React.js, and its component based approach sent a large part of the existing frontend JavaScript frameworks and the web platform itself spinning. Eventually other frameworks like Svelte came along, alongside React meta frameworks like Next.js.

One thing they all have in common are heavy build pipelines, and not running the same code you wrote anywhere. Not on the server and certainly not on the client.

CSS

Love the cascade

HTML is the ultimate declarative language for describing content and structure. The browsers makes sure they're forgiving to your mistakes, and they do a *really* good job.

You get all the buzzword for free!

You can even include elements that the browser doesn't know about, and expand them into functional elements using custom elements.

It's also the language that drives everything else on the web, so browsers are incredibly fast at parsing and rendering the HTML that is downloaded from a server somewhere. A lot of HTML elements have also gotten very good at declaratively defining behavior. Just think about how easy it is to build basic clientside form validation for instance? Unless you need to do some custom stuff, you can get far by simply utilizing HTML.

CSS has always been a powerful language, but in recent years it has gained serious super powers!

Combining the newer declarative behaviors available in HTML with the new CSS super powers, we can do some really cool stuff with very little code!
